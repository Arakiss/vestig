# OTLP Export

Export traces to any OpenTelemetry-compatible backend with zero additional dependencies.

## Overview

The **OpenTelemetry Protocol (OTLP)** is the standard wire format for sending telemetry data to observability backends. Vestig ships a built-in OTLP exporter that:

- **Zero dependencies** — uses native `fetch()` instead of the heavy `@opentelemetry/exporter-trace-otlp-http` SDK
- **Batches and retries** — automatically buffers spans and retries with exponential backoff
- **Edge-compatible** — works on Cloudflare Workers, Vercel Edge Functions, and any runtime with `fetch()`
- **Universal backend support** — Jaeger, Honeycomb, Grafana Cloud, Vercel, Datadog, Axiom, and any OTLP-compatible collector

## Quick Start

```typescript
import { registerSpanProcessor, span } from 'vestig'
import { OTLPExporter } from 'vestig/otlp'

const exporter = new OTLPExporter({
  endpoint: 'https://otel.example.com/v1/traces',
  serviceName: 'my-app',
})

registerSpanProcessor(exporter)

// Your spans are now exported to the OTLP endpoint
await span('db:query', async (s) => {
  s.setAttribute('db.table', 'users')
  return await db.select().from(users)
})
```

## SpanProcessor Interface

The `OTLPExporter` implements the `SpanProcessor` interface, which is how Vestig notifies external systems about span lifecycle events.

```typescript
import type { SpanProcessor } from 'vestig'

interface SpanProcessor {
  onStart?(span: Span): void
  onEnd(span: Span): void
  forceFlush?(): Promise<void>
  shutdown?(): Promise<void>
}
```

- **`onStart`** — called when a span begins (optional, used for real-time streaming)
- **`onEnd`** — called when a span finishes (required, this is where batching exporters collect spans)
- **`forceFlush`** — send all buffered data immediately
- **`shutdown`** — flush remaining data and release resources

### Creating a Custom Processor

```typescript
import type { SpanProcessor } from 'vestig'
import { registerSpanProcessor } from 'vestig'

const consoleProcessor: SpanProcessor = {
  onStart(span) {
    console.log(`[span:start] ${span.name}`)
  },
  onEnd(span) {
    const duration = span.endTime! - span.startTime
    console.log(`[span:end] ${span.name} (${duration.toFixed(1)}ms)`)
  },
  forceFlush() {
    return Promise.resolve()
  },
  shutdown() {
    return Promise.resolve()
  },
}

registerSpanProcessor(consoleProcessor)
```

## Graceful Shutdown

Always flush pending spans before your process exits. Undelivered spans in the buffer are lost if the process terminates without flushing.

```typescript
import { shutdownSpanProcessors } from 'vestig'

process.on('SIGTERM', async () => {
  await shutdownSpanProcessors()
  process.exit(0)
})
```

For Next.js apps, place this in your `instrumentation.ts`:

```typescript
import { registerSpanProcessor, shutdownSpanProcessors } from 'vestig'
import { OTLPExporter } from 'vestig/otlp'

export function register() {
  const exporter = new OTLPExporter({
    endpoint: process.env.OTEL_EXPORTER_OTLP_ENDPOINT!,
    serviceName: 'my-app',
  })

  registerSpanProcessor(exporter)

  process.on('SIGTERM', async () => {
    await shutdownSpanProcessors()
    process.exit(0)
  })
}
```

## Advanced

### Conversion Utilities

For advanced use cases, Vestig exports the low-level conversion functions used internally by the exporter:

```typescript
import {
  toOTLPSpan,
  toOTLPAttributes,
  toOTLPAttributeValue,
  toOTLPStatusCode,
  msToNano,
  isoToNano,
} from 'vestig/otlp'
```

- **`toOTLPSpan(span, epochStartMs)`** — convert a Vestig Span to the OTLP wire format
- **`toOTLPAttributes(attrs)`** — convert a `Record<string, unknown>` to OTLP key-value pairs
- **`toOTLPAttributeValue(value)`** — convert a single value to an OTLP attribute value
- **`toOTLPStatusCode(status)`** — convert a Vestig span status to an OTLP status code
- **`msToNano(ms)`** — convert milliseconds to nanosecond string (OTLP timestamp format)
- **`isoToNano(isoString)`** — convert an ISO date string to nanosecond string

## Edge Compatibility

The OTLP exporter works on Edge runtimes — Cloudflare Workers, Vercel Edge Functions, Deno Deploy — because it relies solely on the `fetch()` API rather than Node.js-specific modules like `http` or `net`.

For Edge environments, consider smaller batch sizes and shorter flush intervals since Edge functions have shorter execution windows:

```typescript
const exporter = new OTLPExporter({
  endpoint: process.env.OTEL_EXPORTER_OTLP_ENDPOINT!,
  serviceName: 'edge-api',
  batchSize: 10,
  flushInterval: 1000,
  timeout: 5000,
})
```
